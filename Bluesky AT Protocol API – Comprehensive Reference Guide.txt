Bluesky AT Protocol API – Comprehensive
Reference Guide
This guide provides a detailed reference for Bluesky’s AT Protocol APIs, covering all current functionality.
It is organized into sections for authentication, record management (posts, profiles, follows, likes), feed
reading  and  custom  feed  generation,  moderation  tools,  content  labels/filtering,  lexicon  schemas,
advanced features, and best practices. All endpoints are identified by their Lexicon NSIDs (names like
com.atproto.*  or app.bsky.* ) and are accessed via HTTP XRPC calls (generally at paths like /
xrpc/<NSID> ). Examples are provided in TypeScript using Bluesky’s official @atproto/api  SDK.
Note on API Hosts:  Many public  read-only endpoints can be queried against Bluesky’s AppView service
(https://public.api.bsky.app ) without authentication. Authenticated or user-specific requests
are typically sent to the user’s own PDS (Personal Data Server) host (e.g. https://bsky.social  for
Bluesky) . The client libraries handle service discovery (resolving a user’s handle to their PDS)
automatically.
Authentication
Bluesky supports two authentication mechanisms: a legacy session token  system (using App Passwords
and JWTs) and a newer OAuth 2.1-based system. As of now, most API usage is via the session token
system, but OAuth is the recommended path moving forward .
Session Endpoint Authentication (Legacy):  Clients can log in by calling 
com.atproto.server.createSession  with the user’s handle (or email) and password (often
an App Password  rather than the main password) . This returns a JSON object containing an 
accessJwt  (short-lived access token) and a refreshJwt  (long-lived token) . For example,
using the TypeScript SDK: 
import{BskyAgent }from'@atproto/api' ;
constagent=newBskyAgent ({service:'https://bsky.social' });
awaitagent.login({identifier :'alice.bsky.social' ,password :'app-
password-123' });
// The agent now holds accessJwt & refreshJwt for authenticated requests.
After login, the client includes the Authorization: Bearer <accessJwt>  header on API calls. The
access  token  expires  after  a  few  minutes,  so  the  client  should  periodically  refresh  it  by  calling
com.atproto.server.refreshSession  with the refreshJwt . The refresh endpoint returns a
new access JWT (and potentially a new refresh token). Most SDKs handle token refresh automatically.
The session can be terminated with com.atproto.server.deleteSession  (revoking the tokens).
App Passwords:  It’s recommended to use an App Password (generated in the user’s account settings)
instead of the main account password for API use. App Passwords can be created/revoked by the user12

OAuth 2.1 Authentication:  The AT Protocol implements an OAuth 2.1 flow for third-party apps
to obtain user authorization without handling raw passwords . In the OAuth model, your
app registers (dynamically) via a Client Identifier  and uses standard authorization code flows.
Users will authorize your app via the server’s OAuth page, and your app receives an access
token. This is the preferred approach for new integrations, offering fine-grained scopes and
improved security (the user never gives the app their password) . Bluesky provides OAuth
client libraries for TypeScript ( @atproto/oauth-client-* ) to streamline this process . 
Example:  A web app can initiate OAuth with the user’s handle, which gets resolved to their PDS (using
DNS or .well-known  as per ATProto handle resolution). The user is redirected to their server’s OAuth
consent screen, and after approval the app receives an authorization code to exchange for tokens
. The OAuth token can then be used as a Bearer token on API calls (similar to the JWT). The OAuth
flow is beyond this guide’s full scope, but developers should refer to the [ATProto OAuth spec and
guides] . OAuth supports scopes (e.g. read:posts, write:posts, etc.), and tokens can be revoked or
limited without affecting the user’s primary credentials.

Creating and Managing Posts (Feed Records)
A post in Bluesky is a record of type app.bsky.feed.post . Posting is done via the generic repository
write endpoint  com.atproto.repo.createRecord . To create a post, you supply your repo DID or
handle, the collection NSID, and the record contents. For example, using the TS SDK:
// Create a new post
awaitagent.post({
text:'Hello world! I posted this via the API.' ,
createdAt :newDate().toISOString ()
});
Under  the  hood,  this  calls  com.atproto.repo.createRecord  with  collection:  
"app.bsky.feed.post"  and a JSON record containing the post text and timestamp . The response
returns the URI of the new post ( at://<your DID>/app.bsky.feed.post/<rkey> ) and its CID
(content hash) . For example: 
{
"uri":"at://did:plc:abcd1234/app.bsky.feed.post/3kzxcvy…" ,
"cid":"bafyreih…"
}
You can use the URI to reference the post elsewhere (e.g. in replies or reposts). The rkey (record key)
in the URI is usually an auto-generated TID (timestamp-based ID) since the  app.bsky.feed.post
schema specifies "key": "tid" .
Posts may include optional fields defined in the lexicon, such as: - reply (to make the post a reply to
another post, containing root and parent URIs/CIDs), - embed (to embed media or other records,
e.g. images or quoted posts), and - facets for rich text (links or mention facets). 
These follow the schemas in  app.bsky.feed.post  and related lexicons. For instance, a reply post
will include a reply field with strong references to the parent and root posts in the thread .
Likewise, images are uploaded via com.atproto.repo.uploadBlob  and then included as an embed
in the post record (with alt text) .
To  delete  a  post,  use  com.atproto.repo.deleteRecord ,  providing  the  collection  and  the
rkey of the record to delete. Deletion permanently removes that record from your repo (though
others who have seen it may have cached it or its CID). There is currently no in-place editing of a post’s
content – to “edit” you would delete and repost, or (in the future) utilize an edit history feature if
introduced.15

Profile Records (User Profile)
A  user’s  profile  (display  name,  bio  description,  avatar ,  etc.)  is  stored  as  a  record
app.bsky.actor.profile  in  their  repository.  By  convention,  this  record  has  the  unique  key
"self"  (the lexicon uses a fixed key since one profile per account) . To update your profile, you
replace this record.
The TS SDK provides a convenient method  agent.updateProfile({...})  (or similar) to call the
appropriate  endpoint.  Under  the  hood,  updating  a  profile  can  be  done  with
com.atproto.repo.putRecord  (since  the  record  already  exists)  –  you  supply  the  collection
app.bsky.actor.profile ,  the  rkey: "self" ,  and  the  new  record  value.  The  profile  record
schema includes fields like displayName  (string), description  (string), and optional avatar and
banner (which are references to uploaded image blobs). For example:
awaitagent.api.com.atproto.repo.putRecord ({
repo:myDid,
collection :'app.bsky.actor.profile' ,
rkey:'self',
record:{
displayName :'Alice Wonderland' ,
description :'Exploring AT Protocol' ,
avatar:{cid:'bafy...' ,mimeType :'image/jpeg' }// blob ref from 
uploadBlob
}
});
The getProfile  and getProfiles  endpoints ( app.bsky.actor.getProfile  / getProfiles )
can retrieve profile records (including badges, follower count, etc.). Note that profiles (and all records in
a user’s repo) are public by default – any server or app can fetch another user’s profile record via their
DID or handle .
Follows and Unfollows
“Following” someone is represented by an app.bsky.graph.follow  record in your repo. When you
follow  user  X,  your  PDS  creates  a  follow  record  with  { subject: X's DID, createdAt:  
timestamp } . This is done via com.atproto.repo.createRecord  similar to posting. For example:
awaitagent.api.com.atproto.repo.createRecord ({
repo:myDid,
collection :'app.bsky.graph.follow' ,
record:{subject:'did:plc:xyz...abcdef' ,createdAt :new
Date().toISOString ()}
});
The  returned  URI  might  look  like  at://<your DID>/app.bsky.graph.follow/3kabc123... .
There is typically no additional content in a follow record aside from the DID of the followed user . To
unfollow ,  you  simply  delete  that  follow  record  from  your  repo  using21

Likes and Reposts
“Likes” (upvotes) and “Reposts” (re-blogging someone’s post) are also stored as records in the user’s
repo:
A Like  is an app.bsky.feed.like  record. It contains a subject  field pointing to the URI of
the liked post (and usually the CID of that post for integrity) and a  createdAt  timestamp.
Creating  a  like  uses  com.atproto.repo.createRecord  on  the  app.bsky.feed.like
collection. The SDK offers methods like agent.like(postUri)  internally. To unlike, delete the
like record.
A  Repost  is an  app.bsky.feed.repost  record. It similarly has a  subject  (the URI of the
post being shared) and createdAt . To repost, create this record; to undo, delete it. Reposts
are essentially the “share/retweet” feature, and clients will show reposted content in feeds (often
annotated as “User A reposted User B’s post”).
These records do not contain freeform text – they just reference the target content. They are public and
can be fetched (e.g. there are endpoints to list who liked a post or who reposted it).

Feed Reading and Generation
This section covers how to retrieve timeline feeds and how to produce custom feeds (an advanced
feature introduced by Bluesky). Feed  reading  refers to fetching posts (your home feed, user feeds,
thread view, etc.), while feed generation  refers to creating algorithmic or curated feeds that others can
follow.
Reading Feeds and Posts
Home Timeline:  The endpoint app.bsky.feed.getTimeline  provides the authenticated
user’s home feed (timeline). This is typically a reverse-chronological view of posts from people
you follow (the exact behavior could be influenced by user’s feed preferences). It’s a GET request
that may accept parameters like cursor for pagination and limit. The result includes an
array of feed posts (usually in a wrapped feed item format including posts and maybe reply/
repost indicators). Example call: 19

Each feed post object contains the post record and often embedded viewer state (e.g. whether you’ve
liked or reposted it).
Author’s Feed:  To get all posts by a specific user , use app.bsky.feed.getAuthorFeed . This
takes an  actor parameter (DID or handle) and returns their posts in reverse chronological
order . It’s essentially that user’s personal feed of their own content.
Fetching Specific Posts:  You can fetch specific posts by URI via  app.bsky.feed.getPosts
(which  accepts  a  list  of  post  URIs  and  returns  the  records  plus  embedded  info  like  author
profiles).  To  retrieve  a  post  along  with  its  reply  thread  context,  use
app.bsky.feed.getPostThread . That returns the post and the conversation thread (parents
and replies) around it. This is useful for showing comments/replies.
Feed  Metadata: app.bsky.feed.describeFeedGenerator  provides  information  about
what feed generation services are configured on the server (e.g. the DID and endpoints of the
feed generator service). This is more relevant in a federated context where custom feeds exist –
essentially it helps clients discover where to query for custom feed content.
Likes  and  Reposts  of  a  Post:  The  endpoints  app.bsky.feed.getLikes  and
app.bsky.feed.getRepostedBy  (if implemented in AppView) allow listing which actors liked
or  reposted  a  particular  post  URI.  These  are  useful  for  showing  the  “Liked  by  X  and  Y”  or
“Reposted by…” information. You supply the uri of the post and get back a list of actor profiles
who have a corresponding like or repost record, plus pagination cursors.
Notification  Feeds:  While  not  exactly  “feed”  in  the  timeline  sense,  note  that
app.bsky.notification.listNotifications  exists  to  fetch  your  notification  stream
(mentions, follows, replies, etc.), and  app.bsky.feed.getTimeline  typically excludes your
own notifications.
Pagination:  Many  feed  endpoints  (timeline,  author  feed,  etc.)  use  cursor-based  pagination.  The
response will include a  cursor string when there are more items; pass this  cursor in the next
request to get the next page . The client should not alter other query params when paginating.
Custom Feed Generation (Algorithmic Feeds)
One of Bluesky’s distinctive features is support for  custom feeds  – alternative algorithms or curated
feeds that users can subscribe to. A custom feed is backed by a Feed Generator service (an independent
service that indexes content and picks posts for the feed). The feed is represented by a record of type
app.bsky.feed.generator  in some user’s repo (often the service operator’s repo), which other
users can discover and follow.
Key points in feed generation:
Feed Generator Record:  This record ( app.bsky.feed.generator ) contains metadata like a
URI or DID of the feed service, a display name and description for the feed, and an  algorithm
identifier . By creating this record, a developer “publishes” a feed. Others can then subscribe to it.•

Fetching  a  Custom  Feed:  To  fetch  posts  from  a  custom  feed,  clients  use
app.bsky.feed.getFeed . This takes a feed parameter (the at:// URI of the feed generator
record, or its identifier) and returns a list of feed items. Under the hood, the AppView will proxy
this request to the feed generator service defined by that record. The feed service responds with
a  skeleton feed  – basically a list of post URIs (and maybe CIDs) in the order of the feed. The
AppView then “hydrates” those URIs into full posts before delivering to the client . Thus,
from the client perspective, getFeed  works similarly to getTimeline  (you get a list of posts),
but the contents are determined by a custom algorithm.
Feed Generator Service API:  If you’re implementing a feed generator service, you need to
implement the XRPC method  app.bsky.feed.getFeedSkeleton  on your server . When the
Bluesky  app  calls  getFeed  (as  above),  the  user’s  AppView  will  call  the  feed  generator’s
getFeedSkeleton  endpoint  (using  the  service  DID  in  the  service  header).  The  feed
generator receives query params like  feed (which feed to serve, in case one service hosts
multiple feeds) and pagination params, then should respond with an ordered list of feed item
URIs (and cursors). The AppView will then fetch those posts from the network and return them to
the user . Essentially, the custom feed service’s job is to maintain an index of recent posts (likely
by consuming the Firehose, see below) and to score/select posts for the feed.
How to Build a Feed Generator:  In short, building a custom feed involves: (1)  Ingesting posts  –
typically by subscribing to the global firehose ( com.atproto.sync.subscribeRepos ) to get all new
posts in real-time, then filtering or storing those you need, and (2) Implementing the feed query  – i.e.,
deciding which posts (from your database) to return for the feed when asked. 
Bluesky’s guidance for feed generation is: “For most use cases, we recommend subscribing to the firehose
at com.atproto.sync.subscribeRepos . This WebSocket will send you every record that is published on
the network… Feed Generators do not need to provide hydrated posts, you can index as much or as little of the
firehose as necessary.” . In practice, your feed service might only keep recent posts (e.g. last 48 hours)
in memory or a database, and when getFeedSkeleton  is called, apply your algorithm to pick the top
N posts to return .
Firehose  (Global  Feed  Stream): com.atproto.sync.subscribeRepos  is  the  event  stream  that
emits all repository updates (across all users on a server , or on an aggregator for the whole network).
It’s how a feed service can get every new post, like, follow, etc. The feed generator can ignore records it
doesn’t care about (e.g. only index app.bsky.feed.post  records for a specific feed about posts, or
maybe also app.bsky.graph.follow  if needed). The Firehose stream returns a sequence of commit
events  with  metadata  and  a  list  of  record  operations  (creates,  updates,  deletes)  included  in  each
commit . Each event has a  seq number and the stream supports resumable consumption via
cursors. No auth is required for subscribeRepos  (it’s a public feed of public data).•

Moderation Tools
Bluesky provides a few layers of moderation controls via the AT Protocol APIs: user-level moderation
(blocking and muting other users, which affects your view), reporting content to server moderators, and
administrative moderation (for moderators to take action). This section focuses on the user-level and
reporting aspects, since admin tools are generally not open to third-party apps without special roles.
Blocks (Account Blocking)
A block  is a stronger action than unfollow – it prevents another account from interacting with you or
seeing your content in their feed (and vice versa, on compliant servers/clients). In AT Protocol, blocks
are  implemented  as  public  records  in  the  user’s  repo,  similarly  to  follows .  The  record  type  is
app.bsky.graph.block , and it’s nearly identical in schema to a follow: it contains a subject  (the
DID of the account being blocked) and a createdAt  timestamp . When you block someone via the
client, the app creates a graph.block  record in your repo.
Public vs Private:  It may sound surprising, but block records are public (anyone could see that A has
blocked B by looking at A’s repo) . This is a deliberate design choice: in a decentralized network,
servers need to know your blocks to enforce them (e.g. filtering content). All compliant servers and
clients will use these records to prevent blocked users from interacting (no replies, mentions, follows,
etc., and profiles may appear as blocked) . However , nothing in a blocked user’s own data is deleted –
the enforcement is done by other participants (PDSes, app views, clients) who refuse to relay or display
content that violates a block . In practice, Bluesky’s app UI also prevents you from seeing the
blocked account’s content unless you log out.2527

Effects of Blocking:  The protocol and clients ensure that a blocked user cannot follow you, see your
posts in their home feed, reply/mention you, or otherwise interact . They may still see your public
content by other means (logged out, alternate account) since the data is public, but direct  interaction is
barred. Block relationships are public as a necessary trade-off to allow network-wide enforcement in a
federated system .
Mutes (Hiding Content)
A  mute  is a personal setting to hide an account’s posts from  your  view, without alerting the other
person  and  without  preventing  their  interactions  elsewhere.  Mutes  are  not  represented  as  public
records (because they don’t need to be enforced by others). Instead, the protocol offers endpoints to
mute/unmute actors and these preferences are stored privately (likely on your PDS). The endpoints are:
app.bsky.graph.muteActor  – a procedure (POST)  to mute a given actor (user). You provide
the DID or handle of the actor to mute. After muting, their posts won’t show in your timeline and
you won’t get notifications from them. This call requires authentication and is scoped to your
account’s PDS .
app.bsky.graph.unmuteActor  – POST to remove a mute on an actor .
Muting is client-specific in effect (each user’s client should respect their mute list). Since mutes don’t
produce a record in your repo, they’re not visible to others. Think of it as an entry in your account’s
“muted users” list on the server .
There are also “mute lists” features (e.g. muting all posts with a certain label or from a certain list of
users), but those are part of content filtering and lists, discussed below.

Labels and Content Filtering
Bluesky uses a system of content labels  to categorize and filter content. Labels are pieces of metadata
attached to records or accounts indicating things like “nsfw”, “spam”, “politics”, etc., which can be used
by clients to decide what to show or hide. The AT Protocol defines a flexible, composable labeling
system and a way for labels to be distributed.
Label Basics
A Label  is essentially a small record with: a source DID  (who created the label), a subject  (what is being
labeled – could be a record URI or an actor DID), a value  (the label term, e.g. nsfw or politics ),
and metadata like  createdAt  and optional  negation or expiration . Labels are  separate
from the records they describe; they aren’t fields inside posts, but rather stand-alone assertions that
“Subject X has label Y”. For example, a moderation service might label a particular post as porn or an
account as spam.
Some labels can also be embedded  in records by the author (self-labeling). For instance, a user might
self-label their post as nsfw by including a label in the record (as per lexicon, some record types allow
a  labels field).  But  generally,  labels  are  created  by  labeler  services  (automated  or  human
moderation systems).
Important details from the spec: - Labels exist as self-authenticating objects  (they can be signed by
their creator). The canonical form is binary (CBOR) for signing, but in JSON they appear with similar
fields . - A label’s val (value) is typically a short ASCII string (like a tag). There is intentionally no
strict global ontology defined in the base spec – values are open-ended, but ecosystem conventions are
emerging  (e.g.  nsfw,  self-harm ,  spoiler ,  etc.).  Certain  prefixes  like  ! are  used  for
“behaviors” (e.g. !takedown  indicating content removed) . - Labels can be negated  by setting
neg: true  which means the label is revoking a previous label of the same value  (e.g. a post was
mislabeled and later the labeler wants to withdraw that label).32

Label Distribution and Queries
Labels are typically produced by a  moderation/labeler service  (like the Bluesky Ozone service) and
need to be distributed to PDSes and app views so that content can be filtered appropriately. The
ATProto spec defines two primary endpoints for this:
com.atproto.label.subscribeLabels : a WebSocket event stream  clients (or services) can
subscribe to, in order to receive  new labels in real-time . It works like other XRPC subscription
endpoints – you connect to  /xrpc/com.atproto.label.subscribeLabels  and receive a
continuous  stream  of  label  events.  It  supports  backfill :  you  can  specify  a  cursor  (sequence
number) to get historical labels, even all the way back to 0 to ingest the full label history . This
stream  is  similar  to  the  repo  “firehose”  but  for  labels  specifically.  When  a  label  is  negated
(redacted), the stream will include an event indicating that (often by still sending the negation as
a new label with neg:true ) .
com.atproto.label.queryLabels : a  HTTP GET  query endpoint for looking up labels by
various filters. You can use it to find all labels on a specific subject (e.g. get all labels on a
particular record or account), or do a broad query of recent labels, with pagination . This is
useful if you want to check, for example, what labels exist for a given post (maybe to display
warning UI or blur it). It’s also the fallback for clients that can’t maintain a live subscription.
These endpoints are often implemented by the  labeler service  (not necessarily by every PDS). For
Bluesky, the Ozone moderation service likely hosts these. A PDS can proxy these requests (similar to
how other service endpoints are proxied).
Availability:  Note that not all labeler services might expose these publicly – the spec allows that label
data might be access-controlled . Bluesky’s approach has been to have a centralized moderation
service that does make labels available to app views with appropriate trust.
Using Labels for Filtering
Clients (and app views) use labels to filter content according to user preferences. For example, a post
labeled  nsfw might be hidden behind a content warning or not shown at all if the user’s settings
disallow NSFW content.
Bluesky defines preference records  that capture a user’s content filtering choices. These are accessible
via  app.bsky.actor.getPreferences  and  settable  via  app.bsky.actor.putPreferences .
Preferences include things like  adultContentPref  (a boolean pref for viewing NSFW content) and
label-specific preferences  (e.g. you could have preferences for “show/hide content with label X”)
.  The  lexicon  app.bsky.actor.defs#contentLabelPref  likely  defines  a  structure  for
preferences on specific label values. For instance, a user might say “Blur content with violence  label”
or “Hide content with porn label”.
When you fetch a feed or a profile, the server (AppView) will by default apply some filtering: it might
exclude content with very severe labels (like illegal content) entirely, and for other labels, it might
include a label metadata in the response so the client can decide to blur or hide. The client can also
request specific handling via HTTP headers:

labeling system for other purposes. The ATProto lexicon defines com.atproto.label.applyLabels
(not certain if exposed to clients) which would allow writing labels (e.g. if you build a custom moderation
bot). However , in general third-party app developers won’t be arbitrarily tagging others’ content with
labels unless they run their own moderation service and users opt into it.
Label Summary
Labels are metadata tags  on content/accounts, used primarily for moderation.
They are distributed via subscribeLabels  (real-time stream of labels) and queryLabels
(on-demand lookup) .
Clients should respect labels by filtering content according to user settings. Bluesky’s reference
app uses user prefs to determine if content with a given label is shown, blurred, or hidden.
Special label values like !takedown  or !suspend  are used by servers to indicate content or
accounts that have been removed from the network (e.g. due to ToS violations). If a post is
labeled !takedown  by a trusted labeler , a compliant app view will redact  it entirely (not just
hide – it won’t even appear in results) . The redact flag in the Accept-Labelers header
instructs whether to do this automatically.
Labels are versioned objects (current version = 1) and can be cryptographically signed . This
allows trust: you might trust labels from the did:plc:mod (Bluesky’s mod team) and not from
unknown sources.

Lexicons and Schema Usage
The AT Protocol uses a schema system called Lexicon  to define all APIs (procedures, queries) and data
structures (records, objects). Understanding lexicons is key to using the APIs effectively, especially if you
plan to define your own record types or work at a low level.
What is a Lexicon?  A lexicon is a JSON schema (somewhat similar to JSON Schema spec) with AT
Protocol-specific features. Each lexicon has a  NSID  (Name-space ID, often reverse-DNS format) as its
identifier . For example: - com.atproto.repo.createRecord  is a lexicon ID for an XRPC procedure. -
app.bsky.feed.post  is a lexicon ID for a record type.
Lexicons describe the types (data models) and the RPC methods  (endpoints). There are different lexicon
“primitive” types: - Record  – defines a record object that can live in a repo (with fields, constraints, etc). -
Query  – defines a GET endpoint (no side effects, primarily for retrieval). - Procedure  – defines a POST
endpoint  (can  have  side  effects/changes).  -  Subscription  –  defines  a  continuous  feed  (WebSocket
subscription).49

Firehose API (Repo Subscription Stream)
The Firehose  refers to the real-time feed of all repository events in the network (or on a given server).
We touched on this in the feed generation section:  com.atproto.sync.subscribeRepos  is the
primary endpoint for this. It’s a WebSocket subscription that sends a continuous sequence of events
representing  repo  updates  (commits).  Each  event  can  include  multiple  record  operations  (create/
update/delete). The firehose is used by services that need to maintain an index of everything (e.g.
search index, custom feed generator , data analytics, etc.).
Important details: - No Auth Required:  By default, anyone can subscribe to the global feed of public
data. On the main Bluesky ecosystem,  bsky.social  exposes  subscribeRepos  for at least the
domains it knows about. In a federated future, one might subscribe to multiple servers or a relay that
aggregates  them.  -  Event  Structure:  The  events  are  defined  in  lexicon
com.atproto.sync.subscribeRepos#commit  (and also #handle  for handle changes, #info for
heartbeat/info messages, etc.). A #commit  event includes the repo DID, a seq number , a commit
CID, an optional prev (prev commit), and crucially an ops list detailing each record change .
Each op has a action (create/update/delete), collection , rkey, and possibly the record value
for creates/updates or the cid being deleted. There’s also a tooBig flag if a commit had too many
changes to send inline, in which case you’d need to fetch the remainder via sync (the spec advises
commits  should  generally  be  small) .  -  Backpressure  &  Resume:  The  subscription  allows  a
cursor query param to resume from a certain sequence. If your client disconnects, you can reconnect
with  ?cursor=<last seen seq>  to  continue.  Typically,  a  firehose  client  should  process  events
sequentially and keep track of the last seq processed. -  Validation:  The protocol expects consuming
services (like relays or app views) to validate the events – e.g. verify signatures on commits, ensure the
DID’s data matches the origin, ignore events for users who have moved servers, etc. . If you’re
building something like a relay or full indexer , consult the “Firehose Validation Best Practices” .
For a simpler consumer (like just populating a feed or logging posts), you might not implement all
validation, but be aware of potential anomalies. -  Use Cases:  Custom feeds (as discussed),  analytics
(e.g. measuring activity), backup  (one could record all events to reconstruct state), or building a search
engine  or aggregator  that needs all data.6364

DID Resolution and Handle System
AT  Protocol  separates  identity  (DIDs  and  handles)  from  servers.  Each  user  has  a  stable  DID  (e.g.
did:plc:abcd1234  – PLC stands for Personal Data Locker , a specific blockchain for DIDs, or could be
did:web:username.com ). They also have a human-readable handle  (like alice.bsky.social  or a
domain name they own). The mapping between handle and DID can change if a user moves to a
different server , but the DID remains the primary key for identity and is what’s used in all records and
API calls .
Resolving  a  Handle  to  a  DID:  To  get  the  DID  for  a  handle,  use
com.atproto.identity.resolveHandle . For example, resolving  "alice.bsky.social"  might
return  did:plc:1234567... . This endpoint is useful because in some contexts (like browser-
based apps) you can’t do DNS lookups directly, so the PDS offers an API call to do it . Under the
hood, resolution follows the rules in the Handle spec: - First, check DNS: look for a TXT record at
_atproto.<handle>  with did=<DID> . - If not found or handle is too long for DNS, fallback to
an HTTPS fetch at  https://<handle>/.well-known/atproto-did  which should contain the DID
. - Then confirm that the DID Document for that DID contains the handle (to avoid spoofing) .
The  resolveHandle  endpoint automates this and returns the DID quickly (it might not do the full
bidirectional check for speed, but generally the expectation is the handle is valid or the DID Document
would be updated accordingly) .
DID Document and Service Discovery:  Once you have a DID (like  did:plc:abc... ), the client or
PDS can resolve the DID document (via the PLC directory or DID method). The DID Document will tell
you the DID’s  public key  (for verifying signatures) and also the URL of the user’s PDS (the service
endpoint) . For  did:plc  (Bluesky’s own DID method), there’s a PLC registry on a blockchain; for
did:web:example.com , the DID document is basically hosted at  https://example.com/.well-
known/did.json . The ATProto Identity spec covers this in detail .
So, if a user enters “alice.bsky.social” into your app: 1. Call  resolveHandle(handle)  → returns
did:plc:xyz .  2.  Then  maybe  call  com.atproto.identity.resolveIdentity  (if  available)  or
directly use the DID to find their PDS. In many cases, your BskyAgent  can be initialized with just the
handle and password; it will internally resolve the DID and PDS. 3. All subsequent requests for that
user’s data (especially writes) should go to the PDS from the DID Document (for bsky.social  users,
that’s https://bsky.social  itself).
Handle updates:  If a user changes their handle (e.g. they bought a domain and want to use that), the
DID stays the same but now resolves to a new handle. The protocol emits handle change events via
subscribeRepos  (a  handle event  type).  Also,  app.bsky.actor.searchActors  and  similar
endpoints can help find users by handle or display name.
DIDs  in  URIs:  All  AT  Protocol  repository  URIs  use  the  DID,  not  the  handle:  e.g.  at://
did:plc:abcd123/app.bsky.feed.post/abcdef123  identifies  a  post  in  that  DID’s  repo.  This
ensures the link remains valid even if the user changes handle or moves servers (as long as the DID
resolution is updated to the new server).7172

com.atproto.sync.requestCrawl  – ask a crawler service to crawl a repo (likely for federation,
when a server learns about a new user or after downtime). - com.atproto.sync.subscribeRepos  –
(the firehose we discussed).
These endpoints support building a full data replication mechanism. If you run a PDS or an indexing
node, you’d use them to keep mirrored copies of user data. For app developers, these are usually not
needed unless you’re doing something like exporting user data or building an app view from scratch.
Direct Messages (Chats)
Note: Direct messaging (DMs) on Bluesky was introduced in 2024 and is still evolving. As of now, DMs – also
referred to as “Chats” – are one-to-one private conversations. They are not stored in your public repo; instead,
Bluesky uses a central service for handling DMs.
Architecture:  DMs use a separate lexicon namespace:  chat.bsky.* . The reason is that DMs likely
involve encryption and privacy considerations that don’t fit the public repo model. Bluesky’s approach is
to route chat messages through a central chat service  (with DID did:web:api.bsky.chat ), while
still using ATProto XRPC calls to interact with it . The user’s PDS acts as a proxy to this service.
Key endpoints in the Chat API (as of now): -  chat.bsky.convo.create  – possibly to create a new
conversation (or this might happen implicitly on first message). - chat.bsky.convo.sendMessage  –
send  a  message  in  a  conversation  (or  start  a  new  convo) .  -
chat.bsky.convo.listConversations  –  list  your  DM  conversation  threads.  -
chat.bsky.convo.listMessages  –  get  messages  in  a  conversation  (with  pagination).  -
chat.bsky.convo.muteConvo  / unmuteConvo  – mute notifications from a particular conversation.
- There might also be endpoints for things like adding a member (if group chats become a thing) but
currently DMs are likely only 1:1.
Using the Chat API:  From a client perspective, to send a DM: 1. Authentication:  You must be logged in
(obviously, since it’s user-specific). 2. Locate the conversation:  If you have an existing conversation ID
(maybe the DID of the other user or a conversation identifier), you use that. If not, the first message
might create the convo. 3. Call sendMessage:  e.g. 82

possibly by calling the same createReport  endpoint on the message URI (if DMs have URIs) or on
the user with a reason indicating it’s about a DM.

Error Handling and Best Practices
To make your integration robust, keep in mind the following:
Standard Error Format:  All XRPC errors come in a standard JSON shape with an  "error"  field (a
short string code) and a  "message"  field (human-readable description) . For example, a request
with  missing  parameters  might  return:  { error: "InvalidRequest", message: "missing  
parameter 'repo'" }  (HTTP status 400). Always check for non-200 HTTP responses and parse the
JSON to get these fields. The error codes are defined per lexicon method (each method can specify error
names). For instance,  authRequired  might be an error for endpoints needing login (HTTP 401),
InvalidToken  for expired JWT, etc. Use these codes to handle specific cases (like prompting re-login
on auth errors).
Rate Limiting:  The API may enforce rate limits. An HTTP 429 error code indicates too many requests.
The  response  might  include  Retry-After  header  (if  not,  you  should  implement  an  exponential
backoff). Best practice is to avoid hammering the API with aggressive polling; instead use the provided
streams (timeline, firehose, etc.) or reasonable intervals.
Data Consistency:  When writing data (creating or deleting records), there is a chance of race conditions
if  multiple  writes  happen  quickly.  The  AT  Protocol  repository  has  an  optimistic  concurrency
mechanism via compare-and-swap . The createRecord , putRecord , and deleteRecord  endpoints
accept optional swapCommit  and/or swapRecord  fields (depending on method) . swapCommit
is a CID of the repo’s current head that you expect; if the repo head has advanced (someone else wrote
something) the operation will fail with a  ConcurrentWriteFailed  error , telling you to refetch the
latest state. This is analogous to an ETag precondition. If you build an app with frequent concurrent
writes (not common for a single-user client, more relevant for multi-device or collaborative scenarios),
consider  using  these  fields  to  avoid  unintentionally  overwriting  data.  The  applyWrites  batch
endpoint can also bundle writes with a single swapCommit  for atomicity .
Retries and Idempotency:  Most record writes are  idempotent  if you use the same record key. For
example, if you try to create a like record and you get a network failure, it’s safe to retry (you might end
up with a duplicate like which isn’t allowed, but the server will likely return an error saying it already
exists). Deleting a non-existent record will similarly error but is harmless. Use client-side UUIDs or
timestamps as needed to ensure idempotency on your side as well.
Time  Format:  Timestamps  in  the  protocol  are  ISO  8601  strings  in  UTC  (e.g.
"2023-11-04T15:20:30.123Z" ).  When  providing  createdAt ,  use
new Date().toISOString() . The servers expect the Z (UTC). If your timestamps are significantly
skewed from real time, some services might reject them (to prevent, say, posts from the far future or
past).84

Content  Sizes:  The  lexicons  often  specify  max  lengths  (e.g.  posts  max  300  characters  or  256
graphemes). The client should enforce these before calling the API to avoid errors. For images, the
app.bsky.embed.images  lexicon says 1MB max per image . Uploading a larger blob will fail with
an error . So check file sizes and possibly downscale images on the client side.
Using the Official SDK vs Raw HTTP:  It’s recommended to use the official @atproto/api  library for
TypeScript/Node if possible. It covers a lot of details: proper headers, session management, serialization
of inputs/outputs to the right types, etc. For example, it will ensure binary blobs are properly sent as
bytes, and will parse the CBOR in binary frames if needed (for subscriptions). If using raw HTTP, make
sure to set Content-Type: application/json  on POSTs (except blob upload which should be the
blob’s mime type). Endpoints under  /xrpc/ don’t do content negotiation – they expect JSON by
default (except subscription which upgrades to WebSocket, and blob download which returns the actual
blob bytes with its mime type).
Testing and Debugging:  Bluesky provides an interactive API docs site (docs.bsky.app) where you can
input  parameters  and  execute  requests  if  you  have  a  bearer  token.  This  can  help  in  seeing  what
responses look like and what errors come back. Also, the @atproto/api  library outputs some useful
info if a request fails (it might throw an error object that includes  error and  message  from the
response).
Versioning:  Keep an eye on Bluesky’s announcements and the bluesky-social/atproto  repo for
changes to lexicons. New fields might appear (clients should ignore unknown fields in responses to be
forward-compatible). Breaking changes are rare, but they might deprecate things like an endpoint in
favor of another (e.g. if in future they unify some API). The lexicon IDs act as version identifiers; a
change usually means a new NSID or a # fragment version bump.
Security:  Use HTTPS for all endpoints. Validate inputs if you accept user text to avoid injecting into API
calls. The AT Protocol uses JWTs for auth which have expiration – store refresh tokens securely if you
need to persist sessions (they are like long-lived credentials). If building a multi-user app, do not mix up
tokens between users – always keep them isolated per user account.
Community and Support:  The AT Protocol is evolving, so some features (like DMs, or future group
interactions) may change or get new lexicons. It’s a good practice to join the developer community
(Bluesky’s dev Slack or forums, etc.) and follow updates. The official documentation at atproto.com  and
docs.bsky.app  is the best source , and the GitHub discussions/issues can also clarify behavior in
edge cases.
With this comprehensive overview, you should be equipped to build on the Bluesky AT Protocol – from
basic posting and social actions, to advanced feed algorithms and moderation-aware client features.
Always refer back to the lexicon definitions for authoritative details, and happy coding with ATProto!
